# CARD-002: Spec-Driven Interview Process for Game Dev

**Status**: DRAFTING  
**Created**: 2026-06-28  
**References**: CARD-001 (MCP tools), CARD-003 (AGENTS.md)

## The Problem

How do we structure conversation-driven spec generation so the AI agent can work autonomously on implementation while staying faithful to user intent?

## Inspiration from infinite-dungeon-game Pattern

The infinite dungeon project uses a "spec-first loop":
1. Human provides natural language direction
2. AI reads existing docs, proposes changes in conversation
3. Human approves or redirects
4. AI updates specs → writes tests → implements code → runs build/test → commits

## Proposed Flow for This Repo

### Phase 0: Initial Interview (One-Shot)

```
Session 1 - Concept Interview
User provides: game concept, genre, scope, constraints
AI produces: docs/spec/overview.md — the single source of truth doc
Includes: core loop, target platforms, art style, tech choices, feature list
Output file structure under docs/specs/
```

### Phase 1: Spec Decomposition

```
Session N+1 - System Specs (per conversation)
User says: "I want a crafting system where players gather materials from the dungeon"
AI reads docs/spec/overview.md → proposes docs/systems/crafting.md spec
User approves or iterates in conversation → spec is finalized
Spec becomes the contract for implementation
```

### Phase 2: Implementation Cycle (Per Spec)

```
1. Agent reads relevant spec document(s) from docs/systems/
2. Agent writes tests in tests/ that verify spec requirements
3. Agent implements code matching spec exactly
4. Agent runs Godot headless test + MCP screenshot verification
5. Agent updates CHANGELOG.md and dev-journal.md
6. Result presented to user: "Spec X implemented. Screenshots attached."
```

### Phase 3: User Review Loop

```
User reviews screenshots/video → says "good" or points at issues
If issues: agent reads spec, finds gap between spec and implementation, fixes it
If good: moves to next spec item from product backlog
```

## Spec Document Templates Needed

| File | Purpose | Author |
|------|---------|--------|
| docs/specs/overview.md | Game concept, scope, core loop | Generated from interview |
| docs/systems/<name>.md | Individual system design spec | AI (human-approved) |
| tests/test_<name>.gd | Spec verification tests | AI agent |
| CHANGELOG.md | What shipped when | AI agent |

## Key Design Decisions

### Where Specs Live vs Code Lives

Specs are in `docs/specs/` and `docs/systems/`. Code is generated by agents into the Godot project structure. The spec documents ARE the source of truth — if code doesn't match spec, that's a bug regardless of what the code says it does.

### Interview Format

The initial interview should be structured but open-ended:
- Genre, core mechanics, target feel (not specific features)
- Scope boundaries (2D vs 3D, single player vs multiplayer, platforms)
- Art direction preferences
- Technical constraints (GDScript vs C#, performance targets)
- Inspiration games and what to borrow/reject from them

### Spec Granularity

Each system spec should be:
- Self-contained enough for an agent to implement without reading other specs
- Specific enough that tests can verify it deterministically
- Short enough (~500-2000 words per spec) that context window limits aren't hit

### MCP Tools During Implementation

When implementing a spec, agents use these Godot AI MCP tools:
1. `godot_inspect_scene` — verify existing scene structure
2. `godot_create_node` / `godot_attach_script` — build scenes programmatically
3. `godot_take_screenshot` — capture visual output for user review
4. `godot_get_parse_diagnostics` — catch syntax errors before running
5. `godot_run_headless_test` — verify logic without UI
